## 摘要

本项目拟基于 `NFL Big Data Bowl` 球员追踪数据构建一个完整的 MLOps 训练与管理流程。项目将以派生的时空监督任务为基础，完成数据清洗、时序切片、特征工程、模型选择、实验追踪、超参数优化、模型注册和模型发布。考虑到项目 2 需要完整展示 `MLflow` 的工程价值，本方案将重点放在“从 tracking 数据到可复现实验”的工程规范上，并在模型层面设计由传统结构化模型到轻量时空深度模型的递进路线。最终输出包括 `MLflow` 实验记录体系、最佳模型、时空样例可视化及可发布的模型说明材料。

关键词: `NFL Big Data Bowl`、`MLflow`、`时空建模`、`轨迹数据`、`MLOps`

## 1. 研究背景与意义

球员追踪数据是一个典型的复杂工程型数据问题。它在数据处理层面包含多表关联、事件对齐和空间归一化，在模型层面包含表格模型、序列模型甚至图模型选择，在发布层面又面临输入结构复杂、推理过程不透明的问题。相比更传统的图像分类题，这类任务更能体现 MLOps 的核心价值:

- 数据版本和任务定义必须严格固化。
- 特征工程与标签工程需要可追溯。
- 不同模型的实验结果必须方便统一比较。

## 2. 项目目标

1. 建立 tracking 数据的可复现处理流程。
2. 使用 `MLflow` 统一记录任务定义、特征版本和模型版本。
3. 形成从结构化 baseline 到轻量序列模型的实验阶梯。
4. 完成最佳模型注册与发布材料整理。

## 3. 研究内容

- 原始 tracking 数据清洗
- 任务定义协议文档化
- 多表 join 与窗口切片
- 特征版本管理
- 模型训练与调参
- 可解释性与样例可视化
- 模型仓库发布

## 4. 模型路线

- baseline:
  - `Logistic Regression`
  - `LightGBM / XGBoost`
- neural:
  - `MLP`
  - `LSTM`
- advanced:
  - 轻量 `Transformer` 或 `Temporal Convolution`

说明:

- 不建议在本课程项目中直接采用过重的图神经网络方案作为主线。
- 可将图模型保留为后续扩展或附加讨论。

## 5. MLflow 方案设计

### 5.1 Run 记录内容

- 任务定义版本
- 时间窗长度
- 特征版本
- 数据抽样策略
- 模型名称
- 超参数
- 主指标与辅指标
- 训练设备信息

### 5.2 Artifact 设计

- 标签协议文档
- 特征字典
- 特征重要性图
- 轨迹可视化图
- 混淆矩阵或误差分析图
- 推理脚本与模型文件

### 5.3 Registry 设计

- `baseline`
- `best-structured`
- `best-sequence`
- `demo`

## 6. 数据与任务版本管理

## 7. 多场景实施方案

### 7.1 方案 A: 稳健课程方案

- 主任务采用结构化分类
- `MLflow` 聚焦特征工程和模型版本管理
- 最佳模型以 `LightGBM` 或 `MLP` 为主

### 7.2 方案 B: 序列增强方案

- 在方案 A 基础上增加 `LSTM` 或 `Transformer`
- 远程 `4090` 负责正式训练
- 最终汇报增加轨迹案例可视化

### 7.3 方案 C: 发布保底方案

- 只发布结构化最佳模型
- 将序列模型结果作为附录或扩展实验

## 9. 可行性与难点

可行性:

- 数据公开可得

难点:

- 发布到 ModelScope 的输入输出说明需要写清楚
