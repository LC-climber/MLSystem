## 1. 本版结论

本轮建议不要选手写体识别、Iris、Titanic 这类模板化题目，优先选择“时序传感器智能识别”方向，兼顾新颖性、系统性和可落地性。

- 主方案: `MLSYS-2026S-P1/P2-HAR`
  - 中文名: `基于多系统的人体活动识别算法对比与 MLOps 发布`
  - 主题: 可穿戴/手机传感器人体活动识别 Human Activity Recognition, HAR
  - 推荐数据集:
    - `PAMAP2`，UCI 官方页给出 `3,850,505` 条实例、`52` 个传感器属性、`18` 类活动，适合做多系统性能与资源对比。
    - `WISDM`，官方页给出 `2,980,765` 条原始样本、`6` 类活动，适合做手机传感器场景补充或第二案例。
- 备选方案: `MLSYS-2026S-P1/P2-AUDIO`
  - 中文名: `基于多系统的城市环境声音识别与 MLOps 发布`
  - 主题: 城市环境音频分类
  - 推荐数据集: `UrbanSound8K`

综合判断，`HAR` 比 `UrbanSound8K` 更适合本课程的“系统对比”要求，因为数据量更大、时序处理更复杂、分布式环节更自然；`UrbanSound8K` 的优势是更有展示效果，适合做现场演示备选。

### 1.1 已确认硬件条件

- 本地硬件:
  - CPU: `Intel Core i5-14600K`
  - 逻辑核心数: `20`
  - 内存: `31 GiB`
  - 当前可用磁盘: `66 GiB`
  - 本地 GPU: 用户确认可用 `RTX 5060 Ti`
- 远程硬件:
  - `单卡 RTX 4090`

据此，本版项目策略调整为:

- 项目 1 的单机版与分布式版主要落在本地 CPU 环境
- 分布式部分默认采用 `Spark 本机伪分布式 / 单机多进程`，报告中明确说明其与真实多机集群的区别
- 项目 2 的深度模型训练与调参主要放到远程 `4090`
- 本地 `5060 Ti` 用于轻量调试、兼容性验证和最终演示保底
- 选题上优先保留“时序信号任务”，因为它更适合 CPU 侧系统对比 + GPU 侧深度模型扩展

### 1.2 深度学习环境约束

- 本地 `RTX 5060 Ti` 需按用户给出的兼容性约束优先配置:
  - `PyTorch >= 2.7.0`
  - `CUDA 12.8` 轮子优先，即 `cu128`
- 推荐环境策略:
  - 远程 `4090` 训练环境与本地 `5060 Ti` 尽量统一为同一主版本栈
  - 优先采用 `torch 2.7.x + torchvision 0.22.x + torchaudio 2.7.x + cu128`
  - 避免在本地使用过旧 `torch 2.6` 及以下版本，减少 `sm_120` 兼容风险

根据 PyTorch 官方“Previous Versions”页面，`PyTorch 2.7.0` 与 `2.7.1` 均提供 `CUDA 12.8` 的官方安装入口，因此把 `2.7.x + cu128` 作为课程项目默认深度学习环境是合理的。

### 3.2 为什么这个题目合适

- 不属于常见课设模板题，较少和“手写数字识别”撞题。
- 既有经典机器学习空间，也有深度学习空间，方便形成两项作业之间的连续叙事。
- 时序传感器数据天然适合讨论数据预处理、滑窗切分、特征工程、模型选择、部署封装等 MLOps 环节。
- 数据规模比很多入门图像/文本数据更适合做系统资源对比。
- 可以做演示: 输入一段传感器序列，输出“走路、跑步、坐着、上下楼”等类别，展示效果直观。

## 4. 项目 1 第一版方案

### 4.1 目标

在单机版、分布式系统、深度学习系统中，实现相同任务和相同算法家族，对比以下维度:

- 模型指标: `Accuracy`、`Macro-F1`、`Weighted-F1`
- 训练效率: 总训练时间、单 epoch 时间、吞吐量
- 推理效率: 单批次推理时间、平均样本推理时间
- 资源开销: CPU/GPU、内存、存储占用、模型大小
- 工程复杂度: 环境配置复杂度、代码复杂度、扩展性

### 4.2 数据与案例

- 主数据集: `WISDM`
  - 原因: 手机加速度传感器场景更直观，预处理难度更可控，更适合当前“本地 CPU + 远程单卡 4090”资源组合
  - 适合案例 A: 手机传感器多类活动识别
- 扩展数据集: `PAMAP2`
  - 原因: 传感器维度更丰富，适合作为项目 1 的增强案例或项目 2 的后续扩展
  - 适合案例 B: 多传感器活动识别

为了满足“1-2 种相同机器学习算法案例”，建议如下设计:

- 案例 A: `Logistic Regression`
  - 单机: `scikit-learn`
  - 分布式: `Spark MLlib`
  - 深度学习系统: `PyTorch` 中实现单层线性分类器
- 案例 B: `MLP`
  - 单机: `scikit-learn MLPClassifier`
  - 分布式: `Spark MultilayerPerceptronClassifier`
  - 深度学习系统: `PyTorch MLP`

这里的关键不是“谁精度最高”，而是分析在同一任务、同类算法下，不同系统在数据规模、资源占用、扩展性和开发成本上的差异。

### 4.3 系统配置建议

- 单机版:
  - `Python 3.10`
  - `scikit-learn`
  - 面向 CPU 训练
- 分布式版:
  - `Spark 3.5 + PySpark`
  - 默认使用本机伪分布式
  - 若后续能临时拿到额外机器，再扩展为小型集群
- 深度学习系统:
  - `PyTorch 2.x`
  - 训练主环境为远程 `单卡 4090`
  - 本地 `5060 Ti` 负责轻量调试与兼容性验证
  - 默认环境版本建议: `PyTorch 2.7.x + cu128`

### 4.4 对比分析亮点

- 对比“特征工程友好型系统”与“端到端建模友好型系统”的差别。
- 对比在单机硬件受限条件下，分布式框架是否仍然值得引入。
- 分析深度学习系统在简单算法任务上是否存在“框架过重”的问题。
- 分析不同系统在可复现性、日志追踪、模型管理上的差别，为项目 2 铺垫。

## 5. 项目 2 第一版方案

### 5.1 目标

围绕同一 HAR 任务，使用 `MLflow` 贯穿以下全流程:

- 数据预处理
- 特征工程
- 模型选择
- 超参数优化
- 实验评价
- 最优模型注册与发布

### 5.2 模型路线

建议从浅到深做三层模型组，方便体现完整 MLOps 过程:

- Baseline:
  - `Logistic Regression`
  - `MLP`
- 中等复杂度:
  - `1D CNN`
  - `CNN + BiLSTM`
- 进阶模型:
  - `Lightweight Transformer` 或 `Temporal Convolutional Network`

结合当前硬件，建议优先次序如下:

- 必做:
  - `MLP`
  - `1D CNN`
- 选做:
  - `CNN + BiLSTM`
  - `Lightweight Transformer`

这样可以避免在单卡 `4090` 上把时间浪费在过重模型上，同时仍然保留足够的技术深度。

建议进一步拆分环境职责:

- `env-local-gpu`
  - 面向本地 `5060 Ti`
  - 目标: 开发调试、可视化、演示保底
- `env-remote-gpu`
  - 面向远程 `4090`
  - 目标: 正式训练、调参、MLflow 批量实验
- `env-cpu`
  - 面向本地 CPU
  - 目标: 数据预处理、Spark、baseline 对比

本地虚拟环境建议统一采用 `conda` 管理，并单独为本课程创建环境，避免与现有研究环境冲突。当前会话已验证 `conda 25.7.0` 可用。

建议环境命名如下:

- `mlsys_cpu`
  - `scikit-learn`、`pyspark`、数据处理
- `mlsys_gpu_local`
  - 本地 `5060 Ti`
  - `PyTorch 2.7.x + cu128`
- `mlsys_gpu_remote`
  - 远程 `4090`
  - 版本尽量与 `mlsys_gpu_local` 一致

### 5.3 MLflow 使用建议

- `Tracking`
  - 记录数据版本、滑窗长度、步长、标准化方式、标签映射、模型参数、训练曲线、评估指标
- `Artifacts`
  - 保存预处理脚本、特征统计图、混淆矩阵、模型权重
- `Model Registry`
  - 注册 `champion` 模型，保留版本和别名
- `Tags`
  - 统一加上课程、项目、数据集、负责人、是否进入汇报版本等标签

MLflow 官方文档说明，Model Registry 提供模型生命周期管理、版本控制、lineage、alias 和 metadata，适合本课程要求的“回溯”和“迭代”。

### 5.4 发布到 ModelScope 的做法

发布环节建议做成“课程可验收版本”，不追求复杂在线服务，优先保证可复现:

- 输出一个可下载模型仓库
- 提供 `README` / model card
- 提供最小推理脚本
- 给出输入格式说明
- 展示 1-2 个推理样例

根据 ModelScope 官方仓库说明，其生态覆盖 `CV`、`NLP`、`Audio`、`Scientific-computation` 等方向，并支持模型训练、推理和 Model-Hub 交互。这里进一步推断，HAR 这类时序模型可以按“自定义模型仓库”方式发布，但会比标准图像/音频任务多一点封装工作量。

如果团队希望降低 ModelScope 发布风险，可在项目 2 里将主模型换成备选音频方向；但从课程完整性看，主线保持 HAR 更统一。

## 8. 风险与备选

- 风险 1: 分布式资源不足
  - 处理: 已按“本机伪分布式”重写方案，报告中明确说明“系统架构模拟与扩展性分析”
- 风险 2: HAR 的 ModelScope 发布封装略麻烦
  - 处理: 保留 `UrbanSound8K` 备选路线
- 风险 3: 原始时序数据预处理工作量较大
  - 处理: 主数据集先固定为 `WISDM`，先做一个固定滑窗版本，后续再迭代更复杂特征
- 风险 4: 数据泄漏
  - 处理: 使用按受试者切分 subject-wise split，而不是随机切分
- 风险 5: 本地 `5060 Ti` 环境与远程 `4090` 环境版本不一致
  - 处理: 优先统一为 `PyTorch 2.7.x + cu128`，将本地环境作为兼容性基线
- 风险 6: 当前工作会话中 `nvidia-smi` 未验证通过
  - 处理: 先按用户确认的本机 GPU 可用事实进行规划，后续在正式环境搭建阶段单独出一份环境验证便函
- 风险 7: 课程项目环境与现有 `conda` 环境依赖冲突
  - 处理: 课程项目只使用新建的 `mlsys_*` 环境，不复用现有环境
