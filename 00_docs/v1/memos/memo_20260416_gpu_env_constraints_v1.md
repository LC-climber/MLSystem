## 1. 本次便函目的

明确课程项目的 GPU 使用优先级与深度学习环境版本约束，避免后续环境反复修改。

## 2. 当前结论

- 正式训练优先使用远程 `单卡 4090`
- 本地 `RTX 5060 Ti` 作为调试、兼容性验证和演示保底环境
- 本地 `5060 Ti` 优先采用:
  - `PyTorch >= 2.7.0`
  - `CUDA 12.8` 对应轮子，即 `cu128`

## 3. 依据

- 用户已明确说明:
  - 本地 `5060 Ti` 需要注意 `sm_120` 架构兼容性
  - 一般需要 `PyTorch 2.7.0` 及以上版本
  - 最好适配 `cu128`
- PyTorch 官方 `Previous Versions` 页面显示:
  - `2.7.0` 提供 `cu128`
  - `2.7.1` 提供 `cu128`

## 4. 推荐环境矩阵

- `mlsys_gpu_local`
  - 用途: 本地开发调试
  - 建议: `torch 2.7.1 / torchvision 0.22.1 / torchaudio 2.7.1 / cu128`
- `mlsys_gpu_remote`
  - 用途: 远程 `4090` 正式训练
  - 建议: 尽量与本地保持相同主版本
- `mlsys_cpu`
  - 用途: `scikit-learn`、`Spark`、数据预处理
  - 建议: 与 GPU 环境解耦

当前会话已验证本地 `conda` 可用，版本为 `25.7.0`。因此课程项目推荐直接使用独立 `conda` 环境，而不是复用已有研究环境。

## 5. 对项目的影响

- 项目 1 的深度学习系统实现可以先在本地 `5060 Ti` 小规模调通，再转远程 `4090` 跑正式结果
- 项目 2 的 MLflow 实验建议统一使用 `2.7.x + cu128`，减少跨机误差
- 最终演示可保留一套本地 GPU 可运行的小模型配置

## 6. 风险与未决事项

- 风险 1: 本地与远程驱动版本可能不同
- 风险 2: 当前本工作会话中 `nvidia-smi` 未验证通过，说明“Codex 当前 shell”与用户本机实际可用状态之间可能存在差异
- 风险 3: 若复用已有 `conda` 环境，容易与现有项目依赖冲突
- 未决事项 1: 远程 `4090` 机器的 `CUDA` 与驱动版本尚未记录
- 未决事项 2: 是否需要锁定 `2.7.1` 而不是 `2.7.0`
